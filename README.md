# Smart Social Media Business Analytics Dashboard

MCA Major Project built with Python, Streamlit, scikit-learn, SQLite, and IBM Watson.

---

## 1. What's in this folder

```
sentiment_project/
│── app.py                 # Streamlit dashboard (frontend + backend, all in Python)
│── train_model.py         # Cleans data, trains ML models, saves them
│── db_helper.py           # SQLite backend logs & retrieves prediction history (CRUD)
│── watson_helper.py       # IBM Watson NLU integration
│── requirements.txt       # Python dependencies (intentionally unpinned see note below)
│── .gitignore             # Excludes raw dataset & local DB from GitHub
│── data/
│   ├── sentiment140.csv    # <-- Real Kaggle dataset goes here (see step 2)
│   ├── clean_sample.csv    # auto-generated by train_model.py, used for dashboard charts
│   ├── cluster_summary.csv # auto-generated: K-Means cluster themes
│   ├── top_hashtags.csv    # auto-generated: trending hashtags
│   └── predictions.db      # auto-created SQLite database (not committed to Git)
│── models/
│   ├── sentiment_model.pkl
│   ├── vectorizer.pkl
│   ├── kmeans_model.pkl      # Audience Segmentation
│   └── engagement_model.pkl  # Engagement Prediction (simulated target)
```

### Modules included

| # | Module | Technique | Notes |
|---|---|---|---|
| 1 | Sentiment Analysis | TF-IDF + Logistic Regression | Core module, trained on real labels |
| 2 | Audience/Content Segmentation | K-Means clustering (unsupervised) | Groups posts into themes from TF-IDF vectors; results are precomputed at training time, not re-clustered live |
| 3 | Trend/Hashtag Detection | Regex extraction + frequency count | Surfaces trending hashtags/keywords |
| 4 | Engagement Prediction | Linear Regression | Trained on a **simulated** engagement score (Sentiment140 has no real likes/shares) — disclosed in the report and in the app itself |
| 5 | Watson Comparison | IBM Watson NLU API | Side-by-side validation against your own model |
| 6 | Prediction History | SQLite (RDBMS) | Logs every live prediction; Create/Read/Delete demonstrated |

### Xquik export support

The batch analysis page also accepts Xquik tweet/search CSV exports. Use any of
these text columns: `text`, `tweet_text`, `full_text`, `content`, or `body`.
Optional metadata such as `created_at`, `query`, `username`, `author_username`,
and `url` is preserved in the downloaded result file.

## 2. Get the real dataset

1. Go to: https://www.kaggle.com/datasets/kazanova/sentiment140
2. Click **Download** (free Kaggle account required) ~240MB zip
3. Unzip it. You'll get a file like `training.1600000.processed.noemoticon.csv`
4. Rename it to `sentiment140.csv` and place it inside `data/`
5. `SAMPLE_SIZE = 50000` in `train_model.py` keeps training fast on a balanced sample of the full 1.6M-row dataset — raise it if you want, but 50k is enough for a solid, credible report.

## 3. Run it locally

```bash
pip install -r requirements.txt
python train_model.py        # trains + saves all models (a few minutes on real data)
streamlit run app.py         # or: python -m streamlit run app.py
```

The app now resolves model and data files relative to `app.py`, so the same
commands work from the project root and from Streamlit deployment runners.

**Windows note:** if `streamlit run app.py` gives `CommandNotFoundException`, use `python -m streamlit run app.py` instead — this avoids a PATH issue common on Windows installs.

**requirements.txt is intentionally unpinned** (`>=` not `==`). Exact-pinned versions caused real build failures during this project — `matplotlib` failed on Windows (no compiler for the pinned version), and `pandas`/`pillow` failed to build on Streamlit Cloud when it defaulted to a very new Python version. Unpinned versions let `pip` pick whichever release already has a ready-made install package for the current environment, avoiding both problems.

## 4. Set up IBM Watson (for the "Watson Analytics" subject tie-in)

1. Go to https://cloud.ibm.com/registration and create a free **Lite** account
2. In the IBM Cloud catalog, search **"Natural Language Understanding"** and create a Lite (free) instance
3. Open the service → **Manage** tab → copy your **API Key** and **URL**
4. Set them as environment variables before running locally:

   **Windows (PowerShell):**
   ```powershell
   $env:WATSON_APIKEY="your_api_key_here"
   $env:WATSON_URL="your_service_url_here"
   ```
   **Mac/Linux:**
   ```bash
   export WATSON_APIKEY="your_api_key_here"
   export WATSON_URL="your_service_url_here"
   ```
5. On the deployed app, add the same two values under **Streamlit Cloud → Settings → Secrets** instead of environment variables.
6. Go to the **"Watson Comparison"** page in the app to test it.

If not configured, the app still runs fine and just shows "Watson not available" on that one page.

---

## 5. Deploy to the cloud

1. Push this folder to a GitHub repository (the `.gitignore` already excludes the large raw dataset and local database — only the trained `.pkl` models are needed for deployment)
2. Go to https://share.streamlit.io and sign in with GitHub
3. Click **"New app"**, select your repo, branch `main`, main file `app.py`
4. Click **"Advanced settings"** before deploying and **set Python version to 3.12**.
   This matters: this project hit a real deployment failure when Streamlit Cloud defaulted to Python 3.14, because the `pillow` package (a Streamlit dependency) had no pre-built install package for that version yet and failed trying to compile from source. Pinning to 3.12 avoids this entirely.
5. If using Watson, add `WATSON_APIKEY` and `WATSON_URL` under **Secrets**
6. Click **Deploy**

**If a later `git push` is rejected** ("fetch first" error): run `git pull` then `git push` again — this resolves the common case where GitHub has a change (e.g. an auto-added file) that your local repo doesn't have yet.

**If the live app shows stale data after a redeploy:** Streamlit's `@st.cache_data` can hold onto old data across a rebuild. Fix with the app's **"⋮" menu → Reboot app**, or **Clear cache** from within the running app.
